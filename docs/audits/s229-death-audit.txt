PART D — DIAGNOSIS / GAP
================================================================================
This is kickoff **outcome (a)**: Zernio ALREADY returns followers (in
`data_raw.accounts[].followersCount`); the edge fn simply never maps them and
hardcodes null. Risk #1 (followers not exposed / need Graph API) is
**DISPROVEN** — no new external API, no scope change, no validator gate.

GBP: the googlebusiness ACCOUNT is present in `accounts[]` (and in
settings.zernio_accounts), so a `google_business` block CAN be surfaced.
However: (1) Zernio's GBP `followersCount` is null at source; (2) GBP
engagement/reach is post-level only and there are no GBP posts in-window, so
GBP engagement/reach stay 0 until GBP posts exist. Account-level GBP
impressions/views are NOT in this Zernio response. So "GBP block appears" is
achievable (presence + null followers + 0/0 until posts) but GBP rich metrics
remain post-driven — documented, not a code bug.

Field naming: actual emitted shape is {facebook, instagram, youtube,
google_business} (via FROM_ZERNIO). The kickoff's "{fb,ig,yt,gbp}" is an
inaccurate recollection of the S225 contract. Consumers (SocialAnalyticsTile,
S228 Dashboard Social mini-tile) read the ACTUAL {facebook,…,google_business}
shape. PER LOCKED DECISION / Risk #3: do NOT rename. No consumer migration.

CURRENTLY vs REQUIRED (the precise mapping change for Phase 2):
  CURRENTLY: data built only from posts[]; followers := null always;
             platforms with no posts omitted entirely.
  REQUIRED:  also iterate data_raw.accounts[]; for each account map
             FROM_ZERNIO[account.platform] → ensure data[key] exists and set
             data[key].followers = account.followersCount ?? null. Platforms
             with an account but no posts get a data entry with followers set
             and engagement/reach = 0. GBP (google_business) thereby appears
             with followers:null, engagement/reach 0 until GBP posts exist.
  Minimal diff to the normalization block (lines ~162-181). No endpoint change,
  no second call (outcome (a), NOT (b)/(c)). verify_jwt stays true; no _shared
  changes.
